2. Project Overview

AURA is a real-time, voice-to-voice conversational AI assistant. The user speaks; the assistant listens, thinks, and replies in natural speech — typically in 1.0–1.5 seconds voice-to-voice (measured and displayed live in the UI for every turn). A four-layer fallback engine guarantees the assistant never goes silent, never shows a raw error, and never abruptly ends the conversation, even when the AI service is slow or down.

4. How Each Requirement Is Met

RequirementImplementation
Accept queries through voice input Browser-native speech recognition with live interim transcription — the transcript is ready the instant the user stops speaking.
Respond with voice output Browser-native speech synthesis; the reply is spoken sentence-by-sentence while the rest is still being generated.
End-to-end response within 2 seconds Streaming pipeline (details in §5); real measured latency is displayed on every turn — typically 1.0–1.5 s.
Graceful fallback for delays / interruptions Four-layer "never go silent" engine (details in §6): spoken fillers, silent retry, offline mini-brain, auto-recovery.
Offline preferred; online acceptable if not feasible Development hardware could not run local LLM inference at conversational speed, so the design maximises the local surface (speech-to-text, text-to-speech and the fallback brain all run in-browser) and uses the fastest available cloud LLM only for response generation — explicitly permitted by the brief.
No generic errors / abrupt endings Every failure path produces a natural spoken response and an invitation to continue the conversation.
Responsiveness, UX, natural conversation Conversation mode (mic auto-reopens after each answer), tap-to-interrupt, voice-reactive UI states, live latency meter.

5. Latency Design (< 2 seconds)

  • Interim speech recognition — transcription happens while the user talks, so there is ~0 delay after they finish.
  • Groq inference — first token typically arrives in 0.3–0.6 s.
  • Sentence-level streaming TTS — the first complete sentence is spoken immediately while later sentences are still streaming; the app never waits for the full reply.
  • Prompt engineering — the system prompt forces a short first sentence (<12 words) so speech can begin as early as possible.
  • Honest measurement — the UI stamps every reply with the true voice-end → voice-start latency.

6. Fallback Design — "Never Go Silent"

LayerTriggerBehaviour
L1Instantly on every queryUI switches to an animated "thinking" state.
L2No token after 1.5 s / 5.5 sSpeaks natural filler phrases ("Hmm, good question, let me think…") to keep the user engaged.
L3Request failsOne silent retry while the filler covers the delay.
L4Retry also fails (offline / outage)A local rule-based mini-brain answers simple intents (time, date, greetings, jokes, small talk) and invites the user to try again — no error messages, no dead ends.
RecoveryNext user turnAutomatically retries the online path and switches back seamlessly.

Known limitation (documented honestly): Chrome's built-in speech recognition itself requires internet, so the fallback layers cover the realistic failure cases — LLM API slowness, outages, rate limits and invalid keys — rather than total network loss.

8. How to Run

  1. pip install -r requirements.txt
  2. cd frontend && npm install && npm run build && cd ..
  3. Copy .env.example to .env and paste a free Groq API key (console.groq.com).
  4. uvicorn server:app --port 8000
  5. Open http://localhost:8000 in Chrome or Edge, tap the mic orb, allow the microphone, and talk.
